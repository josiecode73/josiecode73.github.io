Three.js 產品 360° 展示：從「看起來不錯」到「在 iPhone 跑得動」

J
JosieCode73 · 2026 年 1 月 · 10 min read

ProductOrbit 這個專案一開始讓我很興奮，因為它的功能需求很明確：讓用戶能 360° 旋轉查看 3D 產品，並即時切換顏色和材質。在桌面版做到這個很容易。讓它在 iPhone 12 上也跑得順才是真正的挑戰。

從幾何體開始，不是從 GLTF 開始

做 3D 產品展示，第一個決定是模型的來源：用 Three.js 內建幾何體（BoxGeometry、SphereGeometry、LatheGeometry），還是載入外部 GLTF 模型？

對 ProductOrbit 來說，我選了內建幾何體，主要理由是：

  • 不需要額外的模型資產，初次載入更快
  • 材質切換更直觀（直接改 material，不用處理 GLTF 的材質結構）
  • 幾何體可以程式化生成，針對不同產品形狀靈活調整

如果你的目標是展示真實商品（例如鞋子、包包），GLTF 會更合適。但對於抽象產品展示，內建幾何體搭配好的材質，效果完全夠用。

PBR 材質：三個數值決定一切

PBR（Physically Based Rendering）是讓 3D 物件「看起來真實」的材質系統。Three.js 的 MeshStandardMaterial 就是 PBR 材質，最重要的三個屬性：

const material = new THREE.MeshStandardMaterial({
  color: new THREE.Color('#c0a882'),    // 基礎顏色
  metalness: 0.3,    // 0 = 非金屬, 1 = 純金屬
  roughness: 0.4,    // 0 = 完全光滑, 1 = 完全粗糙
  envMapIntensity: 1.2,    // 環境貼圖反射強度
})

這三個數值的組合決定了物件的「質感感知」。金屬質感：metalness: 0.8, roughness: 0.2；啞光塑膠：metalness: 0, roughness: 0.8；拋光陶瓷：metalness: 0, roughness: 0.1。

💡 envMapIntensity 很重要，但很多教學忘了提。沒有環境貼圖的情況下設這個值沒意義，但一旦加了 HDRI，這個值的高低會戲劇性地改變物件的視覺質感。我通常從 1.0 開始調，需要更強的反射感就往上加。

HDRI 環境貼圖：最有 CP 值的視覺提升

如果只能做一件事來讓 3D 物件「看起來真實」，我會選加 HDRI 環境貼圖。它提供了真實世界的光照資訊，讓物件的反射和光影自動「對」起來。

import { RGBELoader } from 'three/examples/jsm/loaders/RGBELoader'

const loader = new RGBELoader()
loader.load('/hdri/studio-small.hdr', (texture) => {
  texture.mapping = THREE.EquirectangularReflectionMapping
  scene.environment = texture   // 用作環境光源
  scene.background = texture    // 可選：用作背景（通常不用，直接設顏色）
})

HDRI 檔案的大小是一個關鍵考量。4K 的 HDR 檔案可以超過 10MB，對首屏載入很不友善。我用的是 1K 的 HDR（通常 300-500KB），在 Three.js 的渲染尺寸下幾乎看不出差異，但載入速度差很多。

哪裡找合適的 HDRI

Polyhaven 有大量免費的高品質 HDRI，授權是 CC0（完全免費商用）。Studio 類型的 HDRI 對產品展示最合適，中性的打光不會過度干擾物件本身的顏色。

OrbitControls：三個必要的限制

import { OrbitControls } from 'three/examples/jsm/controls/OrbitControls'

const controls = new OrbitControls(camera, renderer.domElement)

// 1. 限制垂直旋轉角度，避免用戶轉到底部看到空白
controls.minPolarAngle = Math.PI * 0.15    // 不能轉太靠近頂部
controls.maxPolarAngle = Math.PI * 0.85    // 不能轉太靠近底部

// 2. 停用 zoom（產品展示通常不需要縮放）
controls.enableZoom = false

// 3. 加入阻尼感，讓旋轉更自然
controls.enableDamping = true
controls.dampingFactor = 0.08

enableDamping 是個很小的改動，但對用戶體驗影響很大。沒有阻尼的 OrbitControls 旋轉是「即停即止」的，有了阻尼以後會有慣性感，整個互動質感提升很多。記得在 animate() 裡每幀呼叫 controls.update()，阻尼才會生效。

行動裝置效能優化：四個有效的策略

在桌面版測試沒問題，上到 iPhone 或低端 Android 就掉幀——這是做 WebGL 最常見的傷心故事。我試過幾個策略，實際有效的是這幾個：

1. 動態 pixel ratio

// 不要直接用 window.devicePixelRatio
// 高 DPR 設備（Retina、高端手機）的 GPU 負擔是 DPR^2
renderer.setPixelRatio(Math.min(window.devicePixelRatio, 2))

把 DPR 上限設在 2，即使是 3x、4x 的 Retina 螢幕也只用 2x 渲染。視覺差異幾乎看不出來，但 GPU 負擔可以少掉很多。

2. 減少幾何體頂點數

桌面版可以用 128 分段的高精度球體，行動版用 32 分段就夠了。差距在近距離才明顯，旋轉查看的時候幾乎無感。用 userAgent 或 navigator.hardwareConcurrency 判斷設備能力，動態選擇幾何體精度。

3. 陰影要謹慎

即時陰影（renderer.shadowMap.enabled = true）是效能殺手之一。如果一定要有陰影感，可以用預先計算的陰影貼圖（baked shadow），或者用一個稍微模糊的橢圓形平面模擬 contact shadow，視覺效果差不多但計算量差很多。

4. requestAnimationFrame 的節流

靜止狀態下不需要每秒 60 幀渲染。我在用戶停止互動 2 秒後，把渲染頻率降到 10fps，有互動時再恢復。這對電池壽命和 CPU 溫度都有幫助。

做 Three.js 最重要的一個心態調整：「在好設備上好看」只完成了一半，「在普通設備上還能用」才是真正的完成。
