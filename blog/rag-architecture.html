AI / RAG

自建 RAG 系統：從 Embedding 到答案生成的全流程，以及那些沒人告訴你的細節

J
JosieCode73 · 2026 年 3 月 · 12 min read

建 AI Assistant 這個專案的時候，我給自己設了一個限制：不能有雲端 API 費用。不是吝嗇，是想弄清楚「純前端的 AI 應用到底能走多遠」。最後做出來的結果讓我有點意外——在瀏覽器端跑推理這件事，比我想的更可行。

這篇把整個 RAG 管線的設計決策都寫出來，包括幾個讓我花了不少時間才搞清楚的地方。

RAG 是什麼，以及為什麼不是 Fine-tuning

RAG（Retrieval-Augmented Generation）的核心思路很簡單：與其讓模型「記住」所有知識，不如在回答問題的時候，先去找相關資料，再把資料連同問題一起交給模型。

Fine-tuning 適合讓模型改變「說話方式」或學習領域特定的格式，但對於「更新知識」這件事效率很差——每次知識庫更新就要重新訓練，成本高而且慢。RAG 則是讓知識庫的更新與模型分離，隨時可以換、隨時可以加。

對我的 use case（讓用戶上傳文件並對其提問）來說，RAG 是唯一合理的選擇。

在瀏覽器裡跑 Transformer：ONNX Runtime Web

這是整個專案最核心的技術選擇。ONNX Runtime Web 讓你可以在瀏覽器裡執行 ONNX 格式的模型，後端加速選項有 WebAssembly 和 WebGPU（目前支援度還不全）。

Embedding 模型我選的是 all-MiniLM-L6-v2 的 ONNX 版本——輸出維度 384，模型大小約 23MB，在效果和體積之間取得了還算合理的平衡。更大的 all-mpnet-base-v2（768 維）效果更好，但 420MB 在瀏覽器裡有點不實際。

import { InferenceSession, Tensor } from 'onnxruntime-web'

// 初始化 session（只做一次，放在 Web Worker 裡）
const session = await InferenceSession.create('/models/all-MiniLM-L6-v2.onnx', {
  executionProviders: ['wasm'],
})

async function getEmbedding(text: string): Promise {
  const tokens = tokenize(text)   // 自行實作或用 transformers.js
  const inputIds = new Tensor('int64', tokens.ids, [1, tokens.ids.length])
  const attentionMask = new Tensor('int64', tokens.mask, [1, tokens.mask.length])

  const output = await session.run({ input_ids: inputIds, attention_mask: attentionMask })
  const embeddings = output['last_hidden_state'].data as Float32Array

  // Mean pooling + L2 normalize
  return meanPoolAndNormalize(embeddings, tokens.ids.length)
}
💡 放在 Web Worker 裡很重要。ONNX 推理會阻塞主執行緒，如果在主執行緒跑，UI 會凍結幾秒。用 Web Worker 隔開，體驗差很多。

Chunking 策略：沒有一個尺寸適合所有人

文件切塊（chunking）是 RAG 裡最容易被低估的步驟。塊太大，向量語意模糊，搜索準確度差；塊太小，上下文不夠，回答品質差。

我試過幾種策略：

  • 固定 token 數（512 tokens，64 overlap）：最簡單，通用性最好，是我最後採用的方案
  • 按段落切：對結構良好的文件效果好，但 PDF 解析出來的文字常常段落判斷很混亂
  • 語意切塊（Semantic chunking）：用相鄰句子的 embedding 相似度決定切點，效果最好但計算成本高

最終我用的是固定 token 數 + overlap，理由很實際：計算成本低，而且對我的 use case（用戶上傳的是各種格式的文件）最穩定。

Tokenization 的細節

不同模型用不同的 tokenizer，計算 token 數的結果會不一樣。all-MiniLM-L6-v2 用的是 WordPiece tokenizer，英文一個詞大約 1-2 個 token，中文則是每個字一個 token。如果你的文件以中文為主，同樣的 chunk size 在字符數上會更長。

向量搜索：不需要向量資料庫

因為是純前端，沒有後端服務，我沒有辦法用 Pinecone 或 Weaviate 這類向量資料庫。替代方案是把 embeddings 存在 IndexedDB 裡，搜索時全部載入記憶體做 cosine similarity 計算。

這個方案的上限大約是幾千個 chunk（幾百頁文件），超過這個規模性能會開始下降。對於大部分的文件 Q&A 場景，這個上限夠用。

function cosineSimilarity(a: number[], b: number[]): number {
  let dot = 0, normA = 0, normB = 0
  for (let i = 0; i < a.length; i++) {
    dot   += a[i] * b[i]
    normA += a[i] * a[i]
    normB += b[i] * b[i]
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB))
}

function searchSimilar(query: number[], chunks: Chunk[], topK = 5): Chunk[] {
  return chunks
    .map(chunk => ({ chunk, score: cosineSimilarity(query, chunk.embedding) }))
    .sort((a, b) => b.score - a.score)
    .slice(0, topK)
    .map(r => r.chunk)
}

Context Window 管理：最容易忽略的環節

找到了相關的 chunk，下一步是把它們拼成 Context 交給語言模型。這裡有一個常見的錯誤：直接把 top-K 的結果全部丟進去。

問題是 Context Window 有長度限制。如果你的模型（我用的是一個本地跑的小型 LLM）Context Window 是 4096 tokens，而你的 top-5 chunk 加起來就 3000 tokens，留給問題和回答的空間就很緊。

我的做法是動態計算：

  1. 先估算問題本身的 token 數
  2. 預留給回答的空間（例如 512 tokens）
  3. 剩餘的空間拿來填 chunk，按相似度分數從高到低填入，超過就停

這樣可以在不超出 Context Window 的前提下，盡量放入最相關的資訊。

關於幻覺（Hallucination）

RAG 能大幅減少幻覺，但不能完全消除。當用戶問的問題在文件裡找不到相關資訊時，模型還是有可能自行「補充」。

我加了一個簡單的 confidence check：如果 top-1 chunk 的相似度分數低於某個閾值（我設的是 0.65），就直接回傳「找不到相關資訊」，不把任何 context 交給模型。

最讓我意外的發現是：chunk 的切法和 Context 的組裝方式，對最終回答品質的影響，比換更大的模型更顯著。在瀏覽器端資源有限的情況下，這兩件事的投資報酬率最高。
