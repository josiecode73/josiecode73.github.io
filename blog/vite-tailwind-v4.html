Vite + Tailwind CSS v4：我踩過的坑與你需要的那份設定

J
JosieCode73 · 2026 年 2 月 · 8 min read

把 monorepo 裡 10 個 app 從 Tailwind v3 全部升上 v4 以後，我最大的感受是：升級這件事本身不難，但設定思路要整個重來。v4 表面上跟 v3 很像，卻在關鍵幾個地方換了玩法。如果你帶著舊直覺走進去，可能會在很奇怪的地方卡住。

這篇不是「照著做就好」的教學，而是我在實際升級過程裡發現的幾個反直覺的地方，加上一份可以直接套用的 monorepo 設定。

CSS-first：最根本的思路轉換

v3 的時代，所有設定都放在 tailwind.config.js（或 .ts）：顏色、間距、字型、plugins 全部在那一份 JS 物件裡。v4 把這個設計翻掉了。

v4 的設定「住」在 CSS 裡，不是 JS 裡。你的主要入口 CSS 長這樣：

@import "tailwindcss";

@theme {
  --color-brand:     #3d63dd;
  --color-brand-dark:#2f52c0;
  --font-sans:       'Inter', system-ui, sans-serif;
  --font-display:    'Lora', Georgia, serif;
  --radius-card:     12px;
  --spacing-section: 64px;
}

定義完以後，這些 token 直接對應到 utility class：text-brand、font-display、rounded-card。不需要在 config 裡特別 extend。

這個轉變比想像中更根本，因為它改變了「設計系統住在哪裡」的問題。以前設計 token 住在 JS，現在住在 CSS。對喜歡 CSS variables 的人來說很直觀，但如果你習慣用 JS 做動態配置（例如根據環境變數切換 theme），需要調整一下思路。

💡 tailwind.config.js 沒有消失，只是變成可選的。如果你有複雜的 plugin 需求或 content 路徑設定，它還是可以用。但基本用法完全不需要它了。

安裝流程（pnpm + Vite，對的那個版本）

v4 的 Vite 整合方式改了。不再透過 PostCSS，改用原生 Vite plugin：

# 注意：tailwindcss@next，直到正式版釋出前
pnpm add -D tailwindcss@next @tailwindcss/vite

然後在 vite.config.ts：

import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  plugins: [
    react(),
    tailwindcss(),   // 就這樣，不需要其他設定
  ],
})

PostCSS 設定可以完全移除。之前的 postcss.config.js 裡的 tailwindcss 和 autoprefixer 都不再需要了——v4 透過 Lightning CSS 處理這些。

⚠️ 如果你的專案還有其他 PostCSS 插件（例如 postcss-import），要確認它們跟 Lightning CSS 模式相容。我遇過一次衝突，症狀是 CSS 變數值被覆蓋，但 build 不報錯，debug 花了一段時間。

Lightning CSS：你不需要懂它，但要知道一件事

Lightning CSS 是 Rust 寫的 CSS 編譯器，v4 預設用它來做 vendor prefix、CSS 嵌套語法展開、以及現代 CSS 特性的 polyfill。速度比 PostCSS 快很多（Rust 嘛），大部分情況下你不需要特別配置它。

但有一件事要知道：Lightning CSS 預設的瀏覽器 target 不一定跟你的 Browserslist 對齊。如果你有明確的瀏覽器支援需求（比如某個企業客戶的 IE11 祖先機台），需要在設定裡明確指定。

Monorepo 下的 @source 指令（這才是重點）

在單一 package 的專案，Tailwind 可以自動找到你的 HTML/TSX 檔案並做 class purging。但在 monorepo 裡，這個自動偵測有時會失效——特別是當你的 UI 元件住在另一個 package，透過 symlink 引用進來的時候。

症狀很詭異：build 沒報錯，但某些 class 在生產環境不見了。開發環境正常是因為 HMR 沒有走 purging。

解法是在 CSS 裡明確宣告 source 路徑：

@import "tailwindcss";

/* 讓 Tailwind 掃描 workspace 裡的其他 package */
@source "../../packages/ui/src/**/*.{tsx,ts}";
@source "../../packages/shared/src/**/*.{tsx,ts}";

@theme {
  /* ... */
}

這個指令告訴 v4 去哪裡找 class 名稱，解決 symlink 環境下掃描不到的問題。

我遇到的三個反直覺問題

1. @theme 不能放在 media query 裡

我試過把部分 token 放在 @media (prefers-color-scheme: dark) 裡來做 dark mode，直接失效。@theme 只能在 CSS 最外層（global scope），不能嵌套。Dark mode token 要用 CSS 變數覆蓋的方式做：

@theme {
  --color-bg: #ffffff;
}

:root[data-theme="dark"] {
  --color-bg: #0f172a;
}

2. JIT 的掃描範圍比 v3 更保守

v3 的 JIT 在 content 路徑設定正確的情況下會大量掃描，v4 更傾向「只掃我看得到的」。動態 class 名稱（比如 `text-${color}` 拼接的字串）一樣不會被掃到，這點跟 v3 一樣——但要更注意 monorepo 跨 package 的情況。

3. 原本用 @apply 的地方可能要重寫

v4 的 @apply 還是可以用，但在某些情況下行為跟 v3 不太一樣，尤其是跟自訂 @theme token 混用時。我有幾個元件的 @apply 在升級後失效，最後直接改成 CSS 變數就解決了。

值得升嗎？

對於新專案：直接用 v4，沒有理由用 v3 了。

對於現有專案：值得升，但不要在衝刺期做。給自己一兩天時間，先在一個小 app 上試跑，確認所有 class 在 build 後都正常，再推到其他 package。

體感上，本地的 HMR 快了 30% 左右，build time 差不多，但 CSS bundle size 稍微縮小了（Lightning CSS 的 tree-shaking 比 PostCSS 更精準）。整體開發體驗是好的。

最讓我喜歡 v4 的地方不是速度，是「設計 token 住在 CSS 裡」這件事。設計師看得懂 CSS，但不一定看得懂 JS config。這個改變讓設計與開發的溝通多了一個共同語言。
