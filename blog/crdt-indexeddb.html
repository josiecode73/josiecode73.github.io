CRDT + IndexedDB：為什麼 NoteVault 不用 localStorage，以及這個決定的代價

J
JosieCode73 · 2026 年 2 月 · 9 min read

NoteVault 的目標是「真正的離線優先」——不是「有網路才能用，斷網還行」，而是「斷網跟有網路用起來一模一樣」。這個目標讓我一路從 localStorage 走到 IndexedDB，再走到 Yjs CRDT，每一步的轉換都有具體的原因。

為什麼不用 localStorage

localStorage 的問題有三個，按嚴重程度排列：

  1. 同步 API 阻塞主執行緒。localStorage.getItem() 在 I/O 慢的設備上（某些 Android 機）真的可以讓 UI 明顯卡頓。
  2. 5MB 上限（大部分瀏覽器）。用戶上傳圖片或儲存大量筆記，很快就會碰到這個天花板。
  3. 只能儲存字串。Binary 資料要先 base64 encode，blob 直接放不進去。

IndexedDB 解決了這三個問題：非同步 API、幾乎無限容量（通常是磁碟空間的 50%）、可以儲存任何 JavaScript 物件包括 ArrayBuffer 和 Blob。

IndexedDB 的使用體驗：沒有你想的那麼痛

IndexedDB 的原生 API 確實繁瑣，但在 2026 年你不需要直接用原生 API。idb 這個 wrapper（Jake Archibald 維護）把 IndexedDB 包成 Promise-based 的乾淨介面：

import { openDB } from 'idb'

const db = await openDB('notevault', 1, {
  upgrade(db) {
    const store = db.createObjectStore('notes', { keyPath: 'id' })
    store.createIndex('updatedAt', 'updatedAt')
    store.createIndex('folderId', 'folderId')
  }
})

// 讀寫跟一般 async/await 一樣自然
const note = await db.get('notes', noteId)
await db.put('notes', { ...note, content: newContent, updatedAt: Date.now() })

為什麼需要 CRDT

離線優先的核心問題是：如果用戶在手機和電腦上都修改了同一篇筆記，重新連線以後要以哪個版本為準？

最簡單的策略是 Last-Write-Wins（LWW）：以時間戳較晚的版本為準。這能解決大部分情況，但有一個致命弱點：如果兩個裝置的系統時鐘不同步（這比你想的更常見），或者兩個裝置幾乎同時修改了不同的段落，LWW 會丟失其中一方的修改。

CRDT（Conflict-free Replicated Data Type）的設計目標就是讓「合併」這件事永遠有一個正確、確定的結果，不管修改的順序如何。

策略優點缺點
Last-Write-Wins實作簡單可能丟失資料；依賴時鐘同步
Operational TransformGoogle Docs 的方案，成熟實作複雜；需要中央伺服器協調
CRDT無需中央伺服器；合併永遠收斂實作複雜；資料結構有額外開銷

Yjs：選它的理由

2026 年主流的前端 CRDT 函式庫有兩個：Yjs 和 Automerge。我選 Yjs 的主要理由：

  • Bundle size 更小：Yjs 約 30KB，Automerge v2 約 280KB（WASM 版）
  • 與 ProseMirror / Tiptap 的整合成熟：如果你用富文字編輯器，Yjs 的生態系更完整
  • 效能：在大型文件上，Yjs 的操作速度通常優於 Automerge
import * as Y from 'yjs'
import { IndexeddbPersistence } from 'y-indexeddb'

const ydoc = new Y.Doc()

// 自動把 CRDT 狀態持久化到 IndexedDB
const persistence = new IndexeddbPersistence('note-' + noteId, ydoc)

// 等待 IndexedDB 載入完成
await persistence.whenSynced

// 操作文件內容
const ytext = ydoc.getText('content')
ytext.insert(0, 'Hello, world')

y-indexeddb 這個包幫你把 Yjs 的狀態自動存到 IndexedDB，並在載入時自動恢復。你不需要手動管理同步，只需要操作 YText、YArray、YMap 等 CRDT 資料結構。

Service Worker：讓離線真正成立

IndexedDB 解決了「資料存哪裡」，但如果你的 React app 本身載入不到（網路斷線時 HTML/JS/CSS 都載不下來），一切都是空談。

Service Worker 的工作是把 app shell（HTML、JS bundle、CSS）快取起來，讓它在完全離線的狀態下也能打開。我用的是 Vite 的 vite-plugin-pwa，設定非常少：

// vite.config.ts
import { VitePWA } from 'vite-plugin-pwa'

export default defineConfig({
  plugins: [
    VitePWA({
      registerType: 'autoUpdate',
      workbox: {
        globPatterns: ['**/*.{js,css,html,ico,png,svg}'],
        // 動態快取：API 回應、字型等
        runtimeCaching: [{
          urlPattern: /^https:\/\/fonts\.googleapis\.com/,
          handler: 'StaleWhileRevalidate',
        }]
      }
    })
  ]
})

這個方案的代價

說清楚代價很重要，因為 CRDT 不是免費的午餐：

  • Tombstone 堆積：CRDT 刪除操作不是真的刪除，而是標記（tombstone），長期使用後文件狀態會越來越大
  • 初始化時間：大型文件首次載入時，CRDT 需要重播所有操作歷史，可能有幾百毫秒的延遲
  • 除錯難度高：CRDT 的狀態不是人讀的，出了問題很難直接看到「哪裡錯了」
如果你的應用不需要多設備同步，只是單純要離線讀寫，localStorage 或 IndexedDB + 簡單 JSON 就夠了。CRDT 的複雜度只有在「多個地方可能同時修改同一份資料」的情境下才真正值得。
