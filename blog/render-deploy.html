Render 部署 10 個 React App：monorepo 的正確設定方式，以及 cold start 的對策

J
JosieCode73 · 2026 年 3 月 · 7 min read

把 10 個 app 部署到 Render 的過程，最費時的不是部署本身，而是搞清楚「pnpm monorepo + Render」的正確設定組合。官方文件對 monorepo 的說明有點語焉不詳，踩了幾個坑以後才找到真正有效的方式。

Render 靜態網站（Static Site）的 monorepo 設定

在 Render 建立每個 app 的 Static Site 時，最重要的兩個設定：

Root Directory

把 Root Directory 設成你的 app 目錄（例如 kanban-board），這樣 Render 的 build 環境會以這個目錄為工作目錄。

⚠️ 不要把 Root Directory 設成 repo 根目錄然後在 build command 裡 cd 進去。這樣做會讓 Render 看不到 monorepo 根目錄的 pnpm-workspace.yaml 和 package.json，導致依賴解析失敗。

Build Command

在 Root Directory 設好以後，build command 要回到 repo 根目錄安裝依賴，再 build 特定 app：

cd .. && pnpm install --frozen-lockfile && pnpm --filter kanban-board build

這個 cd .. 很關鍵。它讓 pnpm 在 workspace 根目錄執行 install，正確解析所有 workspace 依賴，然後用 --filter 只 build 目標 app。

Publish Directory

設成 dist（Vite 的預設輸出目錄）。如果你用了自訂 build.outDir，就改成對應路徑。

BASE_PATH 的必要性

Render 的每個 Static Site 都有獨立的 subdomain（例如 kanban-board.onrender.com），所以 app 本身在根路徑（/）運行，不需要設 base path。

但如果你的部署需要讓多個 app 在同一個 domain 的不同子路徑下運行（例如透過 nginx reverse proxy），就必須設 BASE_PATH：

# 在 vite.config.ts 讀取環境變數
export default defineConfig({
  base: process.env.BASE_PATH ?? '/',
})
# Build 時傳入
BASE_PATH=/kanban-board/ pnpm --filter kanban-board build

Render API：程式化觸發部署

10 個 app 手動點「Deploy」這件事做一次還好，但需要批次更新（例如升 React 版本）的時候，Render API 就很有用了。

# 觸發特定 service 的 deploy
curl -X POST \
  -H "Authorization: Bearer $RENDER_API_KEY" \
  "https://api.render.com/v1/services/{serviceId}/deploys" \
  -H "Content-Type: application/json" \
  -d '{"clearCache": false}'

把 10 個 service ID 存成一個列表，寫個小腳本批次觸發，所有 app 就會開始同時部署。Render 的免費方案允許並行部署，付費方案的並行數量更多。

💡 Service ID 在哪裡找：Render 控制台的 service 頁面，URL 裡的 srv-xxxxx 就是 service ID。或者用 Render API 的 GET /services endpoint 列出所有 service。

Cold Start：免費方案的現實

Render 免費方案的 Static Site 沒有 cold start 問題，因為靜態文件隨時都在。但如果你的 app 有後端（Express、FastAPI 等），免費方案在 15 分鐘無流量後會 spin down，下一個請求需要等 20-30 秒才能回應。

幾個常見的對策：

  • Cron job ping：每 10 分鐘用 cron 服務（如 cron-job.org）打一個 health check 請求，讓服務保持活躍。免費但有點 hack。
  • 付費 plan：Render 的 Starter plan（$7/月）可以避免 spin down。
  • 顯示 loading 狀態：在前端顯示「正在喚醒服務...」的訊息，讓用戶知道在等待，比無聲地等 30 秒好很多。

靜態 app 的 health check

對純靜態 app，Render 沒有 cold start，但你可以在 Render 設定一個 health check 路徑，讓監控工具確認 app 是否正常上線：

/* 在 vite.config.ts 加一個簡單的 health 路由 */
/* 或者直接用 public/health.json */
// public/health.json
{ "status": "ok", "version": "1.0.0" }

環境變數管理

10 個 app 各有各的環境變數，Render 的 Environment Groups 功能可以讓你建立共享的環境變數組，讓多個 service 都能用到相同的值（例如 Analytics ID、Feature Flags）。

設定方式：Render 控制台 → Environment Groups → 建立 group → 在各 service 裡 Link 這個 group。更新 group 的值以後，所有 linked service 都會自動觸發重新部署。

整個部署流程設置好以後，從 git push 到所有 10 個 app 上線大概需要 5-8 分鐘（pnpm install 最花時間）。Render 的 cache 機制在 node_modules 沒有變動時可以顯著縮短這個時間，所以儘量用 --frozen-lockfile 確保 lockfile 一致。
